01Overview

ProdX is a small full-stack web app (FastAPI + React) packaged with a complete DevOps pipeline around it. The point of the project isn't the app — it's the path the app takes to get into production: code → CI → container registry → Terraform-provisioned cluster → GitOps reconciler → publicly reachable URL.

The work is split across three GitHub repositories so the boundaries between app code, infrastructure, and cluster state stay clean:

ProdX

FastAPI backend, React + React Flow frontend, Postgres, Redis. Local docker compose stack and the GitHub Actions workflow that builds + publishes images to Docker Hub.

App

ProdXTF

Terraform for the cluster: VPC + GKE + Argo CD + Cloudflare Tunnel. State lives in HCP Terraform; execution runs locally against your gcloud credentials.

Infrastructure

ProdXCD

Argo CD Application manifests + Kustomize/Helm for ProdX. Argo CD watches this repo and reconciles the cluster — CI never kubectl applys anything.

05Run ProdX locally with Docker Compose

Before touching any cloud, verify the app runs on your machine.

cd ProdX
docker compose up --build

First build takes 2–3 minutes (Node + Python image pulls + frontend npm install). On subsequent runs it's seconds. Once compose finishes:

  • Frontend: http://localhost:8080
  • API: http://localhost:8000/health
  • Metrics JSON: http://localhost:8000/api/metrics

The React Flow canvas loads the prodx graph seeded by the API on boot — a small diagram of Browser → Nginx → FastAPI → Postgres / Redis. The right-hand panel polls /api/metrics every 5s and shows status dots for each dependency.

Data persistence. Postgres lives in a named volume (pg_data). docker compose down keeps the data; docker compose down -v wipes it.

06Docker Hub + GitHub CI

Create a Docker Hub access token

  1. Sign in at hub.docker.com.
  2. Account Settings → Security → New Access Token.
  3. Scope: Read, Write, Delete. Description: prodx-ci.
  4. Copy the token — Docker Hub only shows it once.

Add the secrets to the ProdX repo

On GitHub: Settings → Secrets and variables → Actions → New repository secret. Create exactly these names:

NameValue
DOCKERHUB_USERNAMEYour Docker Hub username (e.g. pbdaemon)
DOCKERHUB_TOKENThe access token from above

How the workflow behaves

The workflow at .github/workflows/docker-publish.yml:

  • builds both Dockerfiles in parallel via Buildx with GitHub Actions cache;
  • on pull requests: builds only, doesn't push;
  • on push to main: pushes :main, :latest, and :sha-xxxxxxx;
  • on a tag v1.2.3: pushes :1.2.3 + the SHA tag too.

Trigger the first run by pushing anything (or just an empty commit):

git commit --allow-empty -m "trigger CI"
git push

Watch the run in the Actions tab on GitHub. When it goes green, your images appear at:

  • https://hub.docker.com/r/pbdaemon/prodx-api
  • https://hub.docker.com/r/pbdaemon/prodx-web

07Cloud Identity + GCP setup

The goal of this step is to give Terraform everything it needs to manage GCP for you — including project creation. The only things you do by hand are the ones that genuinely can't be Terraformed:

ResourceTerraformable?Why
GCP Organization❌ NoProvisioned automatically when you set up Cloud Identity for a domain you own. Identity-layer, not a TF resource.
Billing account❌ NoYou enter a payment method in the Console. Free trial credit attaches here.
Project✅ Yesgoogle_project — handled by envs/bootstrap.
API enablement✅ Yesgoogle_project_service — handled by envs/bootstrap.
environment tag + binding✅ Yesgoogle_tags_tag_* — handled by envs/bootstrap.
Cluster, IAM, networking, everything else✅ Yesenvs/dev takes over from here.

7.1 — Claim the free trial credit

If you don't already have a billing account: sign up at cloud.google.com/free and claim the $300 / 90-day trial. The trial creates a billing account; we'll create the project itself with Terraform in step 10.

7.2 — Set up Cloud Identity Free (gives you the GCP Organization)

Cloud Identity Free is HashiCorp-style: free forever, identity-only (no email), up to 50 users. It's the standard way to provision a GCP Org for personal projects.

  1. Open workspace.google.com/.../sku=identitybasic. The URL must include sku=identitybasic — that selects the free SKU.
  2. Business info: any name (e.g. "ProdX"). Country: yours.
  3. Domain: pbcv.dev (or your domain).
  4. Create the admin user — username like admin at your domain (so admin@pbcv.dev). The mailbox won't actually receive mail; Cloud Identity is identity-only.
  5. Sign in to admin.google.com with the new admin account.
  6. Domain verification: Google gives you a TXT record. Add it in Cloudflare → DNS → Records (type TXT, name @, content google-site-verification=...), then click Verify in admin.google.com. Verification usually completes within a minute.

7.3 — Move the billing account into the new org

Your trial billing account was created when you had no org, so it currently lives outside any org. Terraform can use external billing accounts, but the cleanest setup is to move it in:

  1. Sign in to console.cloud.google.com/billing as your original account (the one that created the billing account).
  2. Click the billing account → Account management → Change organization.
  3. Pick your new org (your domain name will appear).
  4. Confirm.

7.4 — Install gcloud, sign in as the org admin, capture the IDs

winget install Google.CloudSDK
gcloud auth login                          # use your new admin@pbcv.dev account
gcloud auth application-default login

gcloud organizations list                  # copy the ID column → this is org_id
gcloud billing accounts list               # copy the ACCOUNT_ID column → this is billing_account

Save those two values somewhere — they go into envs/bootstrap/terraform.tfvars in step 10.

Why both auth commands?

  • gcloud auth login — credentials for the gcloud CLI itself (kubectl, project switches, etc.).
  • gcloud auth application-default login — credentials Terraform's google provider picks up automatically.

Both have to be the org-admin account for the bootstrap apply to be allowed to create projects.

08HCP Terraform setup

"Terraform Cloud" was renamed to "HCP Terraform" in 2024 — same product, same free tier (up to 5 users). We need two workspaces: one for the bootstrap env (creates the GCP project), one for the dev env (creates the cluster).

  1. Sign up at app.terraform.io/signup/account.
  2. Create an organization — e.g. prodx-demonjr. Note the org name.
  3. Create two CLI-driven workspaces:
    • prodx-bootstrap — holds state for the project + tags.
    • prodx-dev — holds state for the cluster + Argo CD + Cloudflare Tunnel.
    For each: type CLI-driven workflow (not VCS, not API).
  4. For each workspace: open it → Settings → General → Execution Mode → Local. Save.

    Why local execution? Remote execution requires uploading GCP + Cloudflare credentials to HCP Terraform. Local execution runs terraform apply on your machine using your gcloud ADC and your terraform.tfvars; only the state file lives in HCP. Simpler to start with — we can switch to remote execution + workload identity federation later.

  5. On your machine, log Terraform into HCP: 
    winget install Hashicorp.Terraform
terraform login
    A browser window opens, you approve, and a token is written to ~/.terraform.d/credentials.tfrc.json.

Wire the backend to your org

Edit both versions.tf files and replace REPLACE_ME_TFC_ORG with your HCP Terraform org name:

# ProdXTF/envs/bootstrap/versions.tf
cloud {
  organization = "REPLACE_ME_TFC_ORG"   ← change to your HCP Terraform org name
  workspaces {
    name = "prodx-bootstrap"
  }
}

# ProdXTF/envs/dev/versions.tf
cloud {
  organization = "REPLACE_ME_TFC_ORG"   ← same org name here
  workspaces {
    name = "prodx-dev"
  }
}

09Cloudflare setup

Add the zone

Your domain (pbcv.dev in this tutorial) must be a zone in your Cloudflare account. If it isn't yet: dashboard → Add a site → pick the Free plan → change nameservers at your registrar to the two CF nameservers shown. Wait until CF reports the zone as Active (usually under an hour).

Get your Account ID

Open any zone in the dashboard. The right sidebar shows Account ID (a 32-character hex string). Copy it.

Create an API token

  1. Go to dash.cloudflare.com/profile/api-tokens.
  2. Create Token → Custom token.
  3. Name: prodx-terraform.
  4. Permissions:
    ScopeResourceAccess
    AccountCloudflare TunnelEdit
    AccountAccount SettingsRead
    ZoneDNSEdit
    ZoneZoneRead
  5. Zone Resources: Include → Specific zone → pbcv.dev.
  6. Continue → Create. Copy the token immediately — Cloudflare only shows it once.

This token is a secret. Don't paste it into Git, screenshots, or chat. It goes into terraform.tfvars, which is gitignored.

10terraform apply: bootstrap

This is the one-time apply that creates the GCP project, enables APIs on it, creates the org-level environment tag (with all four standard values), and binds Development to the new project. State goes to the prodx-bootstrap HCP Terraform workspace.

Fill in the tfvars

cd ProdXTF\envs\bootstrap
copy terraform.tfvars.example terraform.tfvars
notepad terraform.tfvars

The two required values are the IDs you captured in step 7.4:

org_id          = "123456789012"
billing_account = "XXXXXX-XXXXXX-XXXXXX"
# project_id      = "prodx-dev"          # default; must be globally unique
# project_name    = "ProdX Dev"
# environment_value = "Development"

Apply

terraform init
terraform apply

Takes ~60–90 seconds. Outputs include the new project_id — you'll paste this into the dev tfvars next.

Pin gcloud to the new project

Now that the project exists, point gcloud at it (and pin ADC's quota project so you don't get the warning):

gcloud config set project (terraform output -raw project_id)
gcloud auth application-default set-quota-project (terraform output -raw project_id)

You should see no warnings on the gcloud config set project call this time — the environment tag is already bound by Terraform.

11terraform apply: dev

Create your local tfvars file

cd ..\dev
copy terraform.tfvars.example terraform.tfvars
notepad terraform.tfvars

Fill in the three required values — project_id comes from the bootstrap output:

project_id            = "prodx-dev"
cloudflare_api_token  = "your-cloudflare-api-token"
cloudflare_account_id = "your-cloudflare-account-id"

Defaults take care of region (europe-west3), node count, machine type (e2-small spot), Argo CD chart version, etc. Override in the same file if you want different values.

Initialize

terraform init

Downloads the google, kubernetes, helm, cloudflare, and random providers and configures the HCP Terraform backend. Takes 20–60 seconds the first time.

First apply (two stages)

On a fresh state, the kubernetes and helm providers can't be configured until the cluster exists — chicken-and-egg. Do this once:

terraform apply -target=module.network -target=module.cluster
terraform apply

Total time:

  • Stage 1 (VPC + GKE control plane + node pool): 8–12 minutes.
  • Stage 2 (Argo CD Helm chart + Cloudflare tunnel + cloudflared deployment): 3–5 minutes.

Subsequent applies are a single terraform apply and usually finish in under a minute unless you touched the cluster config.

Why two stages only on the first run? After the cluster exists, Terraform can read its endpoint at plan time, the provider config resolves, and helm + kubernetes resources plan cleanly in one pass.

12Verify it works

Get cluster credentials and check the pods

terraform output -raw get_credentials_command | Invoke-Expression

kubectl get nodes
kubectl get pods -A

You should see:

  • 1 node in Ready state.
  • Argo CD pods in the argocd namespace (argocd-server, argocd-repo-server, argocd-application-controller, …).
  • 2 cloudflared pods in the cloudflared namespace.

Confirm the tunnel is connected

kubectl -n cloudflared logs deploy/cloudflared --tail=40

Look for lines like:

INF Registered tunnel connection connIndex=0 ...
INF Registered tunnel connection connIndex=1 ...
INF Registered tunnel connection connIndex=2 ...
INF Registered tunnel connection connIndex=3 ...

Four registrations per pod = healthy tunnel. Cloudflare uses four edge connections per cloudflared instance for redundancy.

Hit the URLs

terraform output public_urls
  • https://argocd.pbcv.dev → Argo CD login page.
  • https://prodx.pbcv.dev → 502 Bad Gateway until ProdXCD is deployed (the service doesn't exist yet).
  • https://prodxapi.pbcv.dev → 502, same reason.

Log into Argo CD

terraform output -raw argocd_initial_admin_secret_command | Invoke-Expression

That prints the bootstrap password. Username is admin. Log in at https://argocd.pbcv.dev.

Right after first login, delete the bootstrap secret (good hygiene):

kubectl -n argocd delete secret argocd-initial-admin-secret

You now have: a real GKE cluster, Argo CD waiting for work, two publicly reachable URLs hidden behind Cloudflare's edge with TLS — no LoadBalancers, no cert-manager, no static GCP credentials anywhere on disk.

13Next: ProdXCD

The cluster is empty except for Argo CD and cloudflared. The ProdXCD repo (still being built) will contain:

  • A root Argo Application ("app-of-apps") pointing Argo CD at this repo.
  • Kustomize/Helm manifests for the ProdX Deployments + Services in the prodx namespace.
  • StatefulSets for Postgres + Redis with PersistentVolumeClaims.
  • Image tags pinned to whatever's in Docker Hub (initially :latest; later, the GitHub Actions workflow bumps these on every main build).

Once ProdXCD is in place and you've added it as an Argo CD Application:

  1. Argo CD detects the manifests, creates the resources.
  2. cluster pulls pbdaemon/prodx-api + pbdaemon/prodx-web from Docker Hub.
  3. cloudflared was already wired to prodx-web.prodx.svc.cluster.local and prodx-api.prodx.svc.cluster.local — so the 502s flip to 200s the moment the pods are Ready.

No DNS changes, no Terraform changes, no new secrets. That's the GitOps payoff.

14Teardown

Daily / between sessions — keep the project, kill the cluster

To preserve free-trial credit while not actively working:

cd ProdXTF\envs\dev
terraform destroy

This removes the GKE cluster, the VPC, the Argo CD release, the Cloudflare Tunnel, and the three DNS records — about 4–8 minutes. State stays in HCP Terraform so the next apply rebuilds everything from the same config. The GCP project and tag binding stay; the project alone costs $0/month.

Full wipe — destroy the project too

Done with the project entirely:

# Dev first (cluster + tunnel)
cd ProdXTF\envs\dev
terraform destroy

# Then bootstrap (deletes the project, but tag key + values stay
# because they have prevent_destroy = true)
cd ..\bootstrap
terraform destroy

If you also want to remove the org-level environment tag (e.g. you're nuking everything and moving on):

# Temporarily remove the prevent_destroy lifecycle blocks in
# envs/bootstrap/main.tf, then:
terraform destroy

Things terraform destroy does NOT clean up:

  • Container images on Docker Hub (delete in the Docker Hub UI if you want).
  • Your HCP Terraform workspaces (delete in each workspace's Settings → Destruction and Deletion).
  • Cloud Identity / your GCP Organization (leave it — it's free and reusable).
  • The Cloudflare zone (it's your domain — leave it).

15Troubleshooting

Cloud Identity domain verification "couldn't find the record"

DNS propagation between Cloudflare and Google's verifier can lag a few minutes. The TXT record name must be the apex (@, which displays as pbcv.dev in CF). Also make sure the Cloudflare record's proxy status is "DNS only" (grey cloud) — Cloudflare doesn't proxy TXT records, but the toggle should be grey. Try Verify again after 60 seconds.

Bootstrap apply fails: Error 403: The caller does not have permission

You're authenticated as the wrong account. gcloud auth list must show your admin@pbcv.dev Cloud Identity user as ACTIVE, not your original Gmail. Switch with gcloud config set account admin@pbcv.dev and re-run both gcloud auth login + gcloud auth application-default login if needed.

Bootstrap apply fails: billing_account ... is not accessible

The billing account hasn't been moved into the org yet, or your admin user doesn't have Billing Admin on it. Check Console → Billing → Account management → Organization, and IAM on the billing account itself.

CI build fails with denied: requested access to the resource is denied

Docker Hub token has the wrong scope, or the secret names in GitHub are wrong. Re-create the token with Read+Write+Delete and double-check the secrets are spelled DOCKERHUB_USERNAME and DOCKERHUB_TOKEN exactly.

terraform apply errors on the helm release: "Kubernetes cluster unreachable"

Classic first-apply chicken-and-egg. Run the two-stage apply once:

terraform apply -target=module.network -target=module.cluster
terraform apply

Error 403: Project ... is not authorized when calling a GCP API

You skipped the gcloud services enable step or your ADC credentials are stale. Re-run:

gcloud auth application-default login
gcloud services enable compute.googleapis.com container.googleapis.com iam.googleapis.com cloudresourcemanager.googleapis.com

Cloudflare API returns Authentication error (code 10000)

Token scopes are missing. Re-issue with the four permissions listed in Step 9 and the explicit zone include.

Tunnel is created but argocd.pbcv.dev returns "DNS_PROBE_FINISHED_NXDOMAIN"

Cloudflare DNS hasn't propagated yet, or the zone isn't Active. Check dash.cloudflare.com — your zone should show "Active" and the DNS tab should list three orange-cloud CNAMEs.

Tunnel returns 502 for prodx.pbcv.dev

Expected until ProdXCD deploys the prodx-web Service. The Argo CD route should still work — if all three 502, the tunnel itself is fine but cloudflared can't reach any in-cluster service (check that the cloudflared pods are in the same cluster and the namespace labels are right).

GKE node won't schedule pods, says Insufficient memory

e2-small has 2 GB total — Argo CD alone uses ~1 GB. If you add ProdX + Postgres + Redis on top, you need a second node or a larger machine. Easy fix: bump cluster_min_nodes = 2 in terraform.tfvars and apply, or change cluster_machine_type to e2-medium.

I want to nuke the state and start over

Don't. Run terraform destroy first so cloud resources go away, then delete the HCP Terraform workspace. Nuking state without destroying resources leaves orphaned GCP infrastructure that keeps billing.
