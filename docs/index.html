01Project summary

About this document. This is the presentation for the project, not a personal tutorial. The live URLs in the header are the deployed instance using my domain (pbcv.dev), GCP project (prodx-dev), Docker Hub namespace (pbdaemon), and GitHub user (DemOnJR) — concrete values shown as a worked example. Anyone reproducing the project substitutes their own equivalents (see section 15). The Git repositories, Terraform code, Helm chart, CI workflow, and tutorial HTML are public and self-contained.

What's in the repos (public) vs what stays local (secret)

All three repos are public so the project can be forked, reviewed, and reproduced. The split between what's safely committed and what stays on your machine is deliberate:

WhatWhere it livesWhy
Terraform .tf code (modules, envs) ✅ in ProdXTF repo It's just configuration; no secrets, fully forkable
terraform.tfvars.example ✅ in repo (placeholders only) Shows what variables exist; never contains real values
terraform.tfvars (real values: billing account, tunnel token, project id) ❌ gitignored — local to your machine *.tfvars + !*.tfvars.example in .gitignore
Helm chart templates + values.yaml ✅ in ProdXCD repo Image names, replica counts, resource limits are configuration
Postgres password ⚠️ plaintext in ProdXCD/charts/prodx/values.yaml Dev-only; Postgres only listens on the cluster network. Planned hardening: External Secrets Operator + HashiCorp Vault.
Docker Hub access token ❌ GitHub repository secret (DOCKERHUB_TOKEN) Each fork sets its own secret in Settings → Secrets and variables → Actions
Cloudflare tunnel JWT ❌ local terraform.tfvars only If it leaks: delete the tunnel in Cloudflare Zero Trust → the JWT dies with it; create a new tunnel + new token
GCP service account keys ❌ none exist by default Local execution uses your gcloud ADC; remote execution would use a key uploaded to HCP Terraform as a sensitive workspace variable (or Workload Identity Federation — no static key)
Argo CD bootstrap admin password ❌ generated by Argo at install Auto-created Secret argocd-initial-admin-secret; rotate via Argo UI on first login, then kubectl delete the bootstrap secret

For the evaluator forking these repos: after git clone, the only files you create yourself are terraform.tfvars in each envs/ directory (copy from the .example files), and the two GitHub repository secrets DOCKERHUB_USERNAME / DOCKERHUB_TOKEN in your fork's Settings. Everything else is already in the repos.

ProdX is a small full-stack web app (FastAPI backend, React frontend) wrapped in a complete DevOps pipeline. The point isn't the app itself — it's the path the app takes from a developer's machine to a publicly-reachable URL: source → CI build → container registry → Terraform-provisioned GKE cluster → GitOps reconciler → Cloudflare Tunnel → live. Every layer is infrastructure-as-code where possible; the few manual steps that exist (claiming a GCP free trial, clicking through Cloudflare Tunnel setup) are documented and bounded.

The work is split across three GitHub repositories so the boundaries between app code, infrastructure, and cluster state stay clean:

ProdX

FastAPI backend, React + React Flow frontend, Postgres, Redis. Local docker compose stack and the GitHub Actions workflow that builds + publishes images to Docker Hub.

App

ProdXTF

Terraform for the cluster: VPC + GKE + Argo CD + Cloudflare Tunnel. State lives in HCP Terraform; execution runs locally against your gcloud credentials.

Infrastructure

ProdXCD

Argo CD Application manifests + Kustomize/Helm for ProdX. Argo CD watches this repo and reconciles the cluster — CI never kubectl applys anything.

05Run ProdX locally with Docker Compose

Before touching any cloud, verify the app runs on your machine.

cd ProdX
docker compose up --build

First build takes 2–3 minutes (Node + Python image pulls + frontend npm install). On subsequent runs it's seconds. Once compose finishes:

  • Frontend: http://localhost:8080
  • API (direct): http://localhost:8000/health, http://localhost:8000/api/metrics
  • API via the nginx proxy in the web container: http://localhost:8080/api/metrics

The React Flow canvas loads the prodx graph seeded by the API on boot — a small diagram of Browser → Nginx → FastAPI → Postgres / Redis. The right-hand panel polls /api/metrics every 5s and shows status dots for each dependency.

Same image, every environment. The published pbdaemon/prodx-web image doesn't bake any API URL at build time. Its nginx config proxies /api/* to the upstream service name prodx-api — which resolves via Docker Compose service-name DNS in local compose, and via Kubernetes Service DNS in the cluster. That's why http://localhost:8080/api/metrics works here and https://prodx.pbcv.dev/api/metrics works in production, with the same exact image and zero env-var overrides.

Data persistence. Postgres lives in a named volume (pg_data). docker compose down keeps the data; docker compose down -v wipes it.

How the workflow behaves

The workflow at .github/workflows/docker-publish.yml:

  • builds both Dockerfiles in parallel via Buildx with GitHub Actions cache;
  • on pull requests: builds only, doesn't push;
  • on push to main: pushes :main, :latest, and :sha-xxxxxxx;
  • on a tag v1.2.3: pushes :1.2.3 + the SHA tag too.

Terraform manages everything we need inside GCP: the project, API enablement, the cluster, IAM, networking. The only things you do by hand are the two that genuinely can't be Terraformed: signing up for the free trial (billing account creation needs a payment method) and authenticating gcloud locally.

Why no GCP Organization? An Org is only required if you want org-level features (org-level IAM policies, shared VPCs, organization tags, etc.). 7.3 — Capture your billing account ID

Copy the ACCOUNT_ID column (format XXXXXX-XXXXXX-XXXXXX) of the row marked OPEN: True. You'll paste this into envs/bootstrap/terraform.tfvars in step 10.

Appendix — full org-based IaC (optional, ~60+ extra minutes)

If you want the full IaC story with an environment tag bound by Terraform: set up Cloud Identity Free for a domain you own, verify the domain via DNS TXT, capture gcloud organizations list's ID, then grant yourself roles/resourcemanager.projectCreator + roles/resourcemanager.tagAdmin + roles/billing.user via gcloud organizations add-iam-policy-binding on the org. Then put org_id back in the bootstrap module and re-add the google_tags_tag_key/value/binding resources. None of this is required for the cluster work that follows.

08HCP Terraform setup

"Terraform Cloud" was renamed to "HCP Terraform" in 2024 — same product, same free tier (up to 5 users).

What is HCP Terraform actually doing for us?

If you've only used Terraform locally before, the natural question is "why do I need an extra service?". For this project, HCP Terraform plays exactly one role in the default (Local execution mode) setup:

  • It stores the Terraform state file — a JSON document that maps every resource in your .tf code to its real-world counterpart (this GCP project, that GKE cluster, those K8s resources, etc.). Without remote state, that JSON lives in terraform.tfstate on your laptop. Lose your laptop, lose your state, and Terraform has no idea what's already provisioned — you'd have to terraform import everything by hand. HCP keeps it safe.

That's the whole job for now. Specifically, HCP is not:

  • Running terraform commands — that's still happening on your laptop (Execution Mode = Local). HCP's runners do nothing.
  • Holding your GCP credentials — they're in your local gcloud ADC, never uploaded.
  • Watching your repo or auto-applying — that would be VCS-driven mode, which we're not using.
  • Doing policy enforcement (Sentinel) or cost estimation — paid-tier features.

What you get for free even in this minimal mode:

  • State locking — only one terraform apply can run at a time per workspace. Prevents corruption if two people apply simultaneously.
  • State versioning — every apply creates a new state version in the UI. You can browse history, see diffs, and roll back if something goes very wrong.
  • Secret-safe variables (if you choose to upload any) — workspace variables marked sensitive are encrypted at rest.
  • A web UI showing your infrastructure — resource list, drift detection, run history.

Upgrade path: when you want CI to run terraform on your behalf, you switch the workspace to Remote execution and upload credentials (or use Workload Identity Federation with GCP — short-lived tokens, no static key). Same workspaces, same state, just a different runner.

Create the two workspaces

This project uses two workspaces because we have two Terraform root configurations (envs/bootstrap and envs/dev), each with independent state. Workspaces in HCP map 1:1 to root configs.

  1. Sign up at app.terraform.io/signup/account.
  2. Create an organization — e.g. prodx-demonjr. Note the org name.
  3. Create two CLI-driven workspaces:
    • prodx-bootstrap — holds state for the project + tags.
    • prodx-dev — holds state for the cluster + Argo CD + Cloudflare Tunnel.
    For each: type CLI-driven workflow (not VCS, not API).
  4. For each workspace: open it → Settings → General → Execution Mode → Local → Save settings.

    Most-missed step. New workspaces default to Remote execution mode even when created as CLI-driven. If you skip the dropdown change, terraform plan errors with "could not find default credentials" because HCP's runners try to use ADC — which only exists on your laptop. The fix is always: change Execution Mode to Local in this exact settings page, then re-run.

    Why local execution? Remote execution requires uploading GCP + Cloudflare credentials to HCP Terraform. Local execution runs terraform apply on your machine using your gcloud ADC and your terraform.tfvars; only the state file lives in HCP. Simpler to start with — we can switch to remote execution + workload identity federation later.

  5. On your machine, install Terraform from HashiCorp's apt repo and log into HCP: 
    # One-time: add HashiCorp's signed apt repo
wget -O- https://apt.releases.hashicorp.com/gpg \
  | sudo gpg --dearmor -o /usr/share/keyrings/hashicorp-archive-keyring.gpg
echo "deb [signed-by=/usr/share/keyrings/hashicorp-archive-keyring.gpg] https://apt.releases.hashicorp.com $(lsb_release -cs) main" \
  | sudo tee /etc/apt/sources.list.d/hashicorp.list
sudo apt update && sudo apt install -y terraform

terraform login
    A browser window opens, you approve, and a token is written to ~/.terraform.d/credentials.tfrc.json.

Can Terraform run in the cloud instead of on my laptop?

Yes. HCP Terraform supports three execution modes; the project ships with the simplest one but switching is one dropdown + uploading credentials.

ModeWhere TF runsWhat HCP needs from youTrigger
Local (default in this project) Your laptop Nothing — local gcloud ADC is used You run terraform apply
Remote, CLI-driven HCP-hosted Linux runner GCP credentials (service account JSON or OIDC), Cloudflare-related env vars if any You run terraform apply — it streams from HCP to your terminal
Remote, VCS-driven HCP-hosted Linux runner Same credentials + a connection to GitHub/GitLab/Bitbucket Git push triggers plan on PRs and apply on merges to main

Switching this project to Remote execution

  1. HCP Terraform → workspace prodx-dev → Settings → General → Execution Mode = Remote → Save.
  2. Workspace → Variables → Add a variable:
    • Key: GOOGLE_CREDENTIALS
    • Value: the JSON contents of a GCP service account key (download from Console → IAM → Service Accounts → Keys → Add Key → JSON). The SA needs roles/owner on the prodx-dev project, or a more minimal set covering compute.* + container.* + iam.*.
    • Category: Environment variable
    • Sensitive: ✅ checked (encrypted at rest, never shown again)
  3. Run terraform apply as before — output now comes from HCP's runners. State stays where it already was.

Hardened variant: GCP Workload Identity Federation — instead of a static SA key, HCP exchanges a short-lived OIDC token for a temporary GCP credential. No static secret to rotate or leak. HashiCorp has a step-by-step guide; the setup is a one-time 15-minute extra but eliminates the SA-key lifecycle entirely.

What you trade for it:

  • ✅ No GCP credentials on your laptop (or any laptop)
  • ✅ Audit log of every run (who, when, what changed)
  • ✅ With VCS-driven mode: git push = automated apply, no human in the loop
  • ⚠️ Each plan/apply waits for a runner to start (~10–30s cold start)
  • ⚠️ Secrets you previously kept local (CF token) live in HCP now — you trust HCP, not just your laptop
  • ⚠️ Iterating on TF code is slower (no instant feedback from local plan)

Why this project ships with Local execution. The capstone scope is one student building the pipeline end-to-end; the credential upload + the slower iteration loop add friction without value at that scale. Remote execution becomes the right answer once more than one person needs to apply, or once you want CI to do it automatically on Git push.

09Cloudflare setup

The dev env runs cloudflared in the cluster but the tunnel, DNS records, and ingress rules are configured in the Cloudflare Zero Trust dashboard. Less IaC than calling the CF API from Terraform — but no CF API token to manage and no chicken-and-egg between tunnel creation and DNS.

9.1 — Add the zone

Your domain (pbcv.dev in this tutorial) must be a zone in your Cloudflare account. If it isn't yet: dashboard → Add a site → pick the Free plan → change nameservers at your registrar to the two CF nameservers shown. Wait until CF reports the zone as Active (usually under an hour).

9.2 — Create a tunnel

  1. Open one.dash.cloudflare.com (Zero Trust dashboard).
  2. Left sidebar → Networks → Tunnels.
  3. Create a tunnel → Cloudflared → name it prodx-dev → Save tunnel.
  4. The next page ("Install and run a connector") shows a docker run command. Copy the long string that comes after --token — that's your tunnel run token. Save it somewhere; we'll paste it into envs/dev/terraform.tfvars in step 11.
  5. You can ignore the rest of that page — Cloudflare expects you to run their docker command manually, but Terraform will deploy a Kubernetes Deployment with the same image and token instead. Click Next at the bottom of the page.

9.3 — Add Public Hostnames

Still inside the tunnel's setup wizard (or later via the tunnel's Public Hostnames tab), add three entries. CF auto-creates the DNS CNAME records for each.

SubdomainDomainService typeURL
argocdpbcv.devHTTPargocd-server.argocd.svc.cluster.local:80
prodxpbcv.devHTTPprodx-web.prodx.svc.cluster.local:80

The argocd hostname will work right after the dev apply finishes. prodx and prodxapi will return 502 (correctly — the services don't exist yet) until ProdXCD deploys those workloads. No DNS changes needed when the services come up; cloudflared retries on every request.

Tunnel token is a secret. Don't paste it into Git, screenshots, or public chat. It goes into terraform.tfvars, which is gitignored. If a token ends up somewhere it shouldn't, delete the tunnel in CF (the token dies with it) and create a new one.

Pre-flight: clear any stale quota project from ADC

If you've used gcloud with a previous project, your ADC may have a quota_project_id pinned that points to a project you've since deleted. When TF tries to bill API calls to that ghost project, every call fails with "project not found". Clear it before init:

# Check first — prints '1' if set, '0' if not
grep -c quota_project_id ~/.config/gcloud/application_default_credentials.json

# If it printed '1', remove the field:
sudo apt install -y jq    # if jq isn't installed (Linux/WSL)
jq 'del(.quota_project_id)' ~/.config/gcloud/application_default_credentials.json \
  > /tmp/adc.json && mv /tmp/adc.json ~/.config/gcloud/application_default_credentials.json

Safe: jq 'del(.quota_project_id)' only removes that one field. Refresh token and account info are untouched. We'll re-pin the quota project to the new prodx-dev right after apply succeeds.

Apply

terraform init
terraform apply

Takes ~3 minutes (project creation is the slow part; API enablement is seconds each, run in parallel). Outputs include the new project_id and project_number.

Common first-apply failures:

  • "could not find default credentials" — the HCP workspace is still on Remote execution. Go back to step 8 and switch Execution Mode to Local.
  • "project not found" on every call — ADC's quota_project_id is pinned to a deleted project. Run the jq del command above.
  • "Permission 'resourcemanager.projects.create' denied" — your gcloud account is associated with a GCP Organization (Workspace, Cloud Identity, or company tenant) that requires projects to be created under it, and you don't have projectCreator on that org. Quick test: gcloud projects create test-$(date +%s) — if that fails too, the issue is account-level, not Terraform. See the appendix at the end of step 7.

Pin gcloud + ADC to the new project

gcloud auth application-default set-quota-project $(terraform output -raw project_id)

gcloud will still print the environment tag warning. Ignore it — see step 7.3 for why. It's purely cosmetic on parentless projects.

First apply (three stages on a fresh state)

Two independent chicken-and-eggs on a fresh state — both verified the hard way by running terraform destroy followed by terraform apply end-to-end. The correct order is:

# Stage 1 — VPC + GKE cluster + node pool (~14 min; control plane is the slow part)
terraform apply -target=module.network -target=module.cluster

# Stage 2a — Argo CD Helm release (installs the argoproj.io CRDs the next step
# needs) + cloudflared Deployment (~8 min; helm wait + cloudflared image pull)
terraform apply -target=module.argocd -target=module.cloudflared

# Stage 2b — the root Argo CD Application that points at ProdXCD (~1 second:
# just a CR creation; the actual workload sync happens inside Argo afterward)
terraform apply

One-liner for the lazy — chain all three with && so a failure in any stage stops the chain. Total ~22 minutes hands-off; safe to walk away:

terraform apply -target=module.network -target=module.cluster -auto-approve \
  && terraform apply -target=module.argocd -target=module.cloudflared -auto-approve \
  && terraform apply -auto-approve

What goes wrong if you skip a stage:

  • Skipping stage 1 — the kubernetes and helm providers try to talk to a cluster endpoint that doesn't exist yet → "Kubernetes cluster unreachable".
  • Skipping stage 2a (going stage 1 → bare apply) — the kubernetes_manifest.prodx_root_app resource validates its apiVersion: argoproj.io/v1alpha1 against the cluster's discovery API at plan time. Argo CD's CRDs aren't installed yet, so the plan errors with "no matches for kind Application in group argoproj.io".

After the first full apply, every subsequent terraform apply is a single bare command and usually finishes in under a minute unless you touched the cluster config.

Why the staged dance only on the first run? After the cluster + helm release exist, every provider can introspect what it needs at plan time and one pass suffices. The chicken-and-eggs are purely a bootstrap concern — they don't recur.

13.0 — Argo CD configuration (set by ProdXTF, not by you)

The argocd module in ProdXTF installs the official argo-cd Helm chart with all defaults, except two intentional overrides — both because we sit Argo CD behind Cloudflare Tunnel:

# ProdXTF/modules/argocd/main.tf — values passed to helm_release
values = [
  yamlencode({
    server = {
      service = {
        type = "ClusterIP"        # default is LoadBalancer; we don't want a $20/mo GCP LB
      }                           # because cloudflared reaches argocd-server in-cluster.
    }
    configs = {
      params = {
        "server.insecure" = "true"  # makes argocd-server speak HTTP on :80 instead of
      }                             # HTTPS-with-self-signed-cert on :443. TLS is already
    }                               # terminated at Cloudflare's edge before the request
  })                                # enters the tunnel, so HTTPS in-cluster would be
]                                   # double-encryption (and break the CF tunnel ingress
                                    # rule unless we set originRequest.noTLSVerify).

Everything else (the controller, repo-server, applicationset-controller, dex, redis, notifications-controller, the CRDs) uses chart defaults. The Helm release is pinned to chart version 7.7.7 in envs/dev/variables.tf — bump that var if you want to upgrade Argo CD.

What you can configure on Argo CD itself after it's up

  • Change the admin password — log in once with the bootstrap password (kubectl -n argocd get secret argocd-initial-admin-secret -o jsonpath='{.data.password}' | base64 -d), top-right "User Info" → Update Password. Then delete the bootstrap secret (kubectl -n argocd delete secret argocd-initial-admin-secret).
  • Add more Applications — write the manifest into ProdXCD (or another repo) and either commit a new Application there for an existing "root" Application to find (app-of-apps pattern), or add another kubernetes_manifest to envs/dev/argocd_apps.tf.
  • Add repo credentials for private Git repos — Argo CD settings → Repositories → Connect Repo. Or persist via a Secret labeled argocd.argoproj.io/secret-type: repository.
  • RBAC / SSO — Argo CD supports OIDC, GitHub, Google, etc. via the chart's configs.cm and configs.rbac values. We don't bother for a single-user dev cluster.

13.1 — The ProdXCD Helm chart

Layout:

ProdXCD/
└── charts/
    └── prodx/
        ├── Chart.yaml                        name: prodx, version: 0.1.0
        ├── values.yaml                       image tags, replicas, storage sizes, resource limits
        └── templates/
            ├── _helpers.tpl                  common labels
            ├── secret-postgres.yaml          POSTGRES_USER/PASSWORD/DB + DATABASE_URL
            ├── postgres-statefulset.yaml     1Gi PVC, standard-rwo, pg_isready probes
            ├── postgres-service.yaml         headless ClusterIP, port 5432
            ├── redis-statefulset.yaml        256Mi PVC, --appendonly yes, ping probes
            ├── redis-service.yaml            headless ClusterIP, port 6379
            ├── prodx-api-deployment.yaml     pbdaemon/prodx-api:latest, /health probes
            ├── prodx-api-service.yaml        ClusterIP, port 8000
            ├── prodx-web-deployment.yaml     pbdaemon/prodx-web:latest
            └── prodx-web-service.yaml        ClusterIP, port 80

Two design choices worth calling out:

  • imagePullPolicy: Always on both Deployments. The chart uses floating :latest tags, so on every pod restart Docker Hub is hit for a fresh manifest. Trade-off: more pulls, but you can ship a new image to Docker Hub and kubectl rollout restart deployment/prodx-api -n prodx picks it up immediately. The cleaner upgrade path (image-digest bumping by CI) is a future enhancement.
  • StatefulSets for Postgres + Redis with PVCs. GKE's default standard-rwo StorageClass provisions a Persistent Disk (pd-standard) per PVC. Data survives pod restarts and node replacements. Roughly $0.04/GB/month for the underlying PDs.

13.2 — The root Argo Application (in ProdXTF)

Argo CD doesn't auto-discover anything. We need one Application resource that says "watch this Git repo, render the chart at this path, sync into this namespace". Terraform creates it via kubernetes_manifest in envs/dev/argocd_apps.tf:

resource "kubernetes_manifest" "prodx_root_app" {
  manifest = {
    apiVersion = "argoproj.io/v1alpha1"
    kind       = "Application"
    metadata   = { name = "prodx", namespace = "argocd",
                   finalizers = ["resources-finalizer.argocd.argoproj.io"] }
    spec = {
      project     = "default"
      source      = { repoURL = "https://github.com/DemOnJR/ProdXCD.git"
                      targetRevision = "main", path = "charts/prodx",
                      helm = { releaseName = "prodx" } }
      destination = { server = "https://kubernetes.default.svc", namespace = "prodx" }
      syncPolicy  = {
        automated   = { prune = true, selfHeal = true }
        syncOptions = ["CreateNamespace=true", "ServerSideApply=true"]
        retry       = { limit = 5, backoff = { duration = "5s", factor = 2, maxDuration = "5m" } }
      }
    }
  }
  depends_on = [module.argocd]
}

Apply with terraform apply in envs/dev. Argo CD detects the Application within ~30s and starts the first sync.

Why this resource lives in ProdXTF, not ProdXCD. The Application is the bootstrap link between the cluster and ProdXCD; it has to come from somewhere outside ProdXCD itself. Putting it in TF keeps the "cluster bootstrap" story end-to-end Terraform-managed. Once it's in place, every future workload can be added by writing a new Argo Application manifest into ProdXCD — no more TF changes needed.

14Teardown

Daily / between sessions — keep the project, kill the cluster

To preserve free-trial credit while not actively working:

# Step 1: delete the prodx namespace directly. The cascade-deletes the PVCs,
# which in turn delete the underlying Persistent Disks (standard-rwo has
# reclaimPolicy: Delete by default). Without this, the PDs orphan at ~$0.04/GB/month each.
# (Tip: deleting the Argo Application alone with `kubectl delete application prodx` is NOT
# reliable -- in practice Argo CD's finalizer doesn't always cascade in time before
# terraform destroy nukes the helm release, leaving the namespace and PVCs orphaned.)
kubectl delete namespace prodx --wait=false
kubectl wait --for=delete namespace/prodx --timeout=180s

# Step 2: terraform destroy the rest of the env
cd ProdXTF/envs/dev
terraform destroy

This removes the cloudflared deployment, the Argo CD release, the GKE cluster, and the VPC — about 4 minutes. State stays in HCP Terraform so the next apply rebuilds everything from the same config. The GCP project stays; the project alone costs $0/month.

Verify no orphan PDs:

gcloud compute disks list --project prodx-dev --filter='name!~"^gke-"'
# Should print "Listed 0 items." -- only node boot disks (gke-*) survive and they
# go away with the cluster.

About the kubectl delete step: if you skip it and go straight to terraform destroy, the cluster's PVs/PDs may survive cluster deletion and keep billing. The Argo Application's resources-finalizer.argocd.argoproj.io ensures Argo CD removes child resources (PVCs included) when the Application is deleted. If you forgot: check gcloud compute disks list --project prodx-dev for orphans and delete them with gcloud compute disks delete <name> --zone europe-west3-a.

Full wipe — destroy the project too

Done with the project entirely:

# Dev first (cluster + tunnel)
cd ProdXTF/envs/dev
kubectl delete namespace prodx --wait=false && kubectl wait --for=delete namespace/prodx --timeout=180s
terraform destroy -auto-approve

# Then bootstrap (soft-deletes the project)
cd ../bootstrap
terraform destroy -auto-approve

30-day project ID reservation. When the bootstrap destroy soft-deletes the project, GCP keeps the project ID reserved for 30 days (so the deletion can be reversed via gcloud projects undelete <id>). During those 30 days you cannot create a new project with the same ID. To re-apply bootstrap during the reservation window, edit envs/bootstrap/terraform.tfvars with a different project_id (e.g. append a suffix: project_id = "prodx-dev2") and remember to update envs/dev/terraform.tfvars to match before applying dev.

Verified the hard way — this document was rebuilt against prodx-dev2 after a clean destroy+replay of bootstrap.

After destroying bootstrap, your ADC's quota_project_id still points at the now-soft-deleted project. Future TF runs will fail with "project not found". Clear it before re-applying:

python3 -c 'import json,os; p=os.path.expanduser("~/.config/gcloud/application_default_credentials.json"); d=json.load(open(p)); d.pop("quota_project_id", None); json.dump(d, open(p,"w"))'
# or with jq:
# jq 'del(.quota_project_id)' ~/.config/gcloud/application_default_credentials.json > /tmp/adc.json && mv /tmp/adc.json ~/.config/gcloud/application_default_credentials.json

Then after the new bootstrap apply succeeds, re-pin: gcloud auth application-default set-quota-project <new-project-id>.

Things terraform destroy does NOT clean up:

  • Container images on Docker Hub (delete in the Docker Hub UI if you want).
  • Your HCP Terraform workspaces (delete in each workspace's Settings → Destruction and Deletion).
  • The Cloudflare zone (it's your domain — leave it).

# Daily: kill the cluster, keep the project (~4 min terraform destroy)

# Full wipe (project too) — see warning below about ID reservation
cd ../bootstrap && terraform destroy -auto-approve

# After bootstrap destroy, clear stale ADC quota_project_id so future TF
# runs against a different project don't hit "project not found":
python3 -c 'import json,os; p=os.path.expanduser("~/.config/gcloud/application_default_credentials.json"); d=json.load(open(p)); d.pop("quota_project_id", None); json.dump(d, open(p,"w"))'

30-day project ID reservation. After bootstrap destroy, the project enters DELETE_REQUESTED state and its ID is reserved for 30 days. You cannot create a new project with the same ID until the reservation expires (or you successfully gcloud projects undelete <id>). To re-apply bootstrap before then, change project_id in envs/bootstrap/terraform.tfvars to a different value (e.g. add a suffix: prodx-dev2) AND update envs/dev/terraform.tfvars to match before re-applying dev. Verified the hard way — this entire document was rebuilt against prodx-dev2 after a clean bootstrap destroy+replay.

Every command above is the same one you'd find scattered through sections 5–14 — this section just consolidates them in execution order with no narrative. Use it as a sanity check when you're stuck on whether you missed a step.

16Troubleshooting

HCP Terraform plan errors with "could not find default credentials"

The workspace is still on Remote execution mode. HCP's runners don't have your local gcloud ADC, so calls error out. Open the workspace → Settings → General → Execution Mode → Local → Save settings, then re-run terraform plan. New workspaces default to Remote even when created as CLI-driven.

Bootstrap apply fails: every API call says "project not found"

Your ADC's quota_project_id is pinned to a deleted project. Check with grep -c quota_project_id ~/.config/gcloud/application_default_credentials.json — if it prints 1, clear the field (see step 10 pre-flight). New gcloud versions auto-pin quota project to whatever was active during application-default login; if that project later got deleted, every TF call breaks.

Bootstrap apply fails: Permission 'resourcemanager.projects.create' denied

Your Gmail/Workspace account is associated with a GCP Organization (visible via gcloud organizations list) and that org requires projects to be created under it. Quick test: gcloud projects create test-$(date +%s). If that also fails, it's an account-level constraint, not a Terraform bug. Options: (a) get roles/resourcemanager.projectCreator on the org, (b) switch to a Gmail account that isn't org-bound, or (c) follow the org-based variant in step 7's appendix.

Bootstrap apply fails: LifecycleState for parent organizations/... is DELETE_REQUESTED

An org you're associated with is being torn down by Google (usually because a Cloud Identity / Workspace subscription was cancelled). Run gcloud organizations describe <ORG_ID> to confirm. There's no quick fix from gcloud — either restore the subscription via admin.google.com within the grace period, or sign in with a different Gmail account that isn't bound to that dying org and re-run gcloud auth login + gcloud auth application-default login.

gcloud config set project warns about a missing "environment" tag

Harmless. Environment tags are org-scoped and parentless projects (which the default no-org bootstrap creates) literally cannot have one bound. There is no action you can take to silence it short of moving to the org-based variant — and it doesn't block anything.

CI build fails with denied: requested access to the resource is denied

Docker Hub token has the wrong scope, or the secret names in GitHub are wrong. Re-create the token with Read+Write+Delete and double-check the secrets are spelled DOCKERHUB_USERNAME and DOCKERHUB_TOKEN exactly.

terraform apply errors on the helm release: "Kubernetes cluster unreachable"

First chicken-and-egg: the helm/kubernetes providers can't reach the cluster until it exists. Do stage 1 of the three-stage first apply (see 15.9):

terraform apply -target=module.network -target=module.cluster

terraform plan errors with "no matches for kind 'Application' in group 'argoproj.io'"

Second chicken-and-egg: the kubernetes_manifest.prodx_root_app resource validates the Argo CD CRDs at plan time, but they only get installed by the helm release in the same configuration. Do stage 2a before the bare apply:

terraform apply -target=module.argocd -target=module.cloudflared
terraform apply

Error 403: Project ... is not authorized when calling a GCP API

You skipped the gcloud services enable step or your ADC credentials are stale. Re-run:

gcloud auth application-default login
gcloud services enable compute.googleapis.com container.googleapis.com iam.googleapis.com cloudresourcemanager.googleapis.com

Cloudflare API returns Authentication error (code 10000)

Token scopes are missing. Re-issue with the four permissions listed in Step 9 and the explicit zone include.

Tunnel is created but argocd.pbcv.dev returns "DNS_PROBE_FINISHED_NXDOMAIN"

Cloudflare DNS hasn't propagated yet, or the zone isn't Active. Check dash.cloudflare.com — your zone should show "Active" and the DNS tab should list three orange-cloud CNAMEs.

Tunnel returns 502 for prodx.pbcv.dev

Expected until ProdXCD deploys the prodx-web Service. The Argo CD route should still work — if all three 502, the tunnel itself is fine but cloudflared can't reach any in-cluster service (check that the cloudflared pods are in the same cluster and the namespace labels are right).

GKE node won't schedule pods, says Insufficient memory

e2-small has 2 GB total — Argo CD alone uses ~1 GB. If you add ProdX + Postgres + Redis on top, you need a second node or a larger machine. Easy fix: bump cluster_min_nodes = 2 in terraform.tfvars and apply, or change cluster_machine_type to e2-medium.

I want to nuke the state and start over

Don't. Run terraform destroy first so cloud resources go away, then delete the HCP Terraform workspace. Nuking state without destroying resources leaves orphaned GCP infrastructure that keeps billing.
