ProdX — Complete Setup Guide

Everything you need to take this project from zero to a live website — written so anyone can follow it, even with no DevOps background.

Plan ~1 hour the first time. Most of that is waiting for the cloud cluster to build.  ·  Curious what each piece is? See the Architecture & Components page.

How to read this guide Do the sections in order, top to bottom. Steps 1–7 get you ready. Step 8 builds everything. Anything in a dark box is a command — copy it (use the Copy button), paste it into your terminal, press Enter. Replace anything written like YOUR-SOMETHING with your own value.

1 · What is ProdX?

ProdX is a small web app (a product catalog) that runs on real cloud infrastructure. The whole thing is split into three GitHub repositories, each with one job:

RepositoryWhat it holdsThink of it as…
ProdXThe application code — the backend API and the website.The product
ProdXTFTerraform code that builds the cloud infrastructure.The construction plan
ProdXCDThe Helm chart describing what runs inside the cluster.The furniture layout

When everything is running you get three public web addresses (on a domain you own):

  • https://prodx.YOUR-DOMAIN — the website
  • https://prodxapi.YOUR-DOMAIN — the backend API
  • https://argocd.YOUR-DOMAIN — the deployment dashboard (Argo CD)
You don't need to understand the jargon Words like Kubernetes, Terraform or Helm are explained in plain English in the Glossary at the bottom. You can finish this guide without reading it — just follow the steps.

2 · Accounts you need

Create these five accounts first. All of them are free to start.

AccountWhat it's forSign up
GitHub Stores the three code repositories. github.com/join
Google Cloud Runs the actual servers (the Kubernetes cluster). cloud.google.com/free
Cloudflare Gives the app real HTTPS web addresses. You also need a domain name added to Cloudflare. dash.cloudflare.com
HCP Terraform Safely stores the "memory" of what infrastructure exists. app.terraform.io
HashiCorp Vault A secure vault that holds passwords the app needs. This project uses a Vault server already running at https://vault.pbcv.dev. Use an existing Vault server (you need a login for it).
Google Cloud — the $300 free trial When you sign up, Google gives you $300 of free credit that lasts ~90 days. It asks for a credit card to verify you (a tiny charge may appear and is refunded). This project costs only a few dollars a month if you remember to pause it when you're not using it — well within the free credit.
About the domain name Cloudflare needs a domain you own (e.g. from Namecheap, GoDaddy, Cloudflare itself…). After buying it, add it to Cloudflare and switch your domain's nameservers to the two Cloudflare gives you. This guide's reference setup uses pbcv.dev — you will use your own.

3 · Install the tools

These are small programs that run on your computer. Open a terminal and install them.

Windows (PowerShell, "Run as administrator")

brew install terraform kubectl helm vault
brew install --cask google-cloud-sdk
gcloud components install gke-gcloud-auth-plugin

Linux / WSL (Ubuntu or Debian)

# Terraform + Vault (from HashiCorp's package repo)
curl -fsSL https://apt.releases.hashicorp.com/gpg | sudo gpg --dearmor -o /usr/share/keyrings/hashicorp.gpg
echo "deb [signed-by=/usr/share/keyrings/hashicorp.gpg] https://apt.releases.hashicorp.com $(lsb_release -cs) main" | sudo tee /etc/apt/sources.list.d/hashicorp.list
sudo apt-get update
sudo apt-get install -y terraform vault

# Then install gcloud, kubectl and helm from their official guides:
#   https://cloud.google.com/sdk/docs/install
#   https://kubernetes.io/docs/tasks/tools/
#   https://helm.sh/docs/intro/install/
gcloud components install gke-gcloud-auth-plugin

Check that the main tools are installed:

4 · Fork the three repositories

"Forking" makes your own copy of a repository on GitHub. Do it for all three:

  1. Open each repo and click the Fork button (top-right):
    • github.com/DemOnJR/ProdX
    • github.com/DemOnJR/ProdXTF

  2. Then download all three to your computer (replace YOUR-GITHUB-USERNAME):
git clone https://github.com/YOUR-GITHUB-USERNAME/ProdX.git

5 · Log in to everything

These commands connect the tools on your computer to your cloud accounts. Each one opens a browser window — approve it and come back to the terminal.

Google Cloud

# 1. Log in as yourself
gcloud auth login

# 2. Give Terraform permission to act on your behalf
gcloud auth application-default login

The first command logs you in. The second creates a separate credential that Terraform uses automatically — you need both.

HCP Terraform

terraform login

Type yes, approve in the browser. This saves a token so Terraform can store its state safely online.

HashiCorp Vault

export VAULT_ADDR=https://vault.pbcv.dev
vault login

On Windows PowerShell use $env:VAULT_ADDR="https://vault.pbcv.dev" instead of the export line. Logging in saves a token to ~/.vault-token that Terraform and the vault command both pick up automatically.

Cloudflare

Cloudflare has no login command — you connect to it with an API token instead. You'll create that token in the next step.

6 · Collect your keys & IDs

Now gather every value the project needs. Keep them in a scratch note — you'll paste them into config files in Step 8.

The full checklist

ValueWhere to get it
Google billing account ID
looks like 01ABCD-234567-89EFGH		 Run gcloud billing accounts list — use the row where OPEN is True.
GCP project ID You invent this. Lowercase letters, digits and dashes, globally unique — e.g. prodx-dev-7421.
HCP Terraform organization name The organization you created at app.terraform.io.
Your domain name The domain you added to Cloudflare — e.g. example.com.

GitHub username Your GitHub handle.

Database password You invent this — pick a strong password for the app's database.

How to create the Cloudflare API token

  1. Make sure your domain is already added to Cloudflare and its nameservers are switched (see the note in Step 2).
  2. Go to dash.cloudflare.com/profile/api-tokens → Create Token.
  3. Start from the "Edit zone DNS" template.
  4. Add two more permissions to that template:
    • Account → Cloudflare Tunnel → Edit
    • Zone → DNS → Edit  (under Zone Resources, limit it to your domain)
  5. Click Continue → Create Token, then copy the token.
Copy it now Cloudflare shows the token's value only once. If you lose it, you must create a new one.

How to create the GitHub access token (PAT)

  1. Go to github.com/settings/tokens?type=beta → Generate new token (fine-grained).
  2. Resource owner: your account. Repository access: Only select repositories → pick ProdX, ProdXTF, ProdXCD.
  3. Set these Repository permissions:
    PermissionAccessWhy
    AdministrationRead and writeBranch protection rules
    ActionsRead and writeCI workflow settings
    VariablesRead and writeSo Terraform can publish the deploy settings
    SecretsRead and writeOnly if you use the optional Cursor review
    ContentsRead-onlyReading the repos
    MetadataRead-onlyGranted automatically
  4. Click Generate token and copy it.

7 · One config edit

Two files have the HCP Terraform organization name written inside them. This is the only hand-edit you can't avoid (Terraform doesn't allow this particular setting to be a variable).

In your ProdXTF folder, open these two files and change the organization line to your HCP org name:

  • envs/bootstrap/versions.tf
  • envs/dev/versions.tf
  cloud {
    organization = "YOUR-HCP-ORG"     # <-- change this line in BOTH files
    workspaces { name = "..." }
  }

8 · Run the environments

Now you build everything. ProdXTF is organized into five environments — each is just a folder you go into and run a couple of commands. Run them in this exact order:

#FolderWhat it buildsTime
1envs/tfc-bootstrapThe online "save slots" for Terraform's state~10 s
2envs/bootstrapThe Google Cloud project~3 min
3envs/devThe cluster + Argo CD + registry (the big one)~22 min
4envs/vault-bootstrapConnects the cluster to Vault, stores the passwords~1 min
5envs/github-bootstrapLocks down the GitHub repos (branch protection)~10 s

Each environment has a file called terraform.tfvars.example. The pattern is always the same: copy it to terraform.tfvars, fill in your values, then apply. Your real terraform.tfvars files are kept private (they're git-ignored) so your keys never end up on GitHub.

What "terraform apply" does terraform init gets things ready. terraform apply shows a plan and asks you to confirm — type yes and press Enter. Terraform then builds the cloud resources for you.

8.1 — tfc-bootstrap

cd ProdXTF/envs/tfc-bootstrap
cp terraform.tfvars.example terraform.tfvars
# edit terraform.tfvars:  organization = "YOUR-HCP-ORG"
terraform init
terraform apply

8.2 — bootstrap (the Google Cloud project)

cd ../bootstrap
cp terraform.tfvars.example terraform.tfvars
# edit terraform.tfvars:
#   billing_account = "01ABCD-234567-89EFGH"
#   project_id      = "YOUR-PROJECT-ID"
terraform init
terraform apply

After it finishes, tell gcloud to use this new project by default:

gcloud config set project $(terraform output -raw project_id)
gcloud auth application-default set-quota-project $(terraform output -raw project_id)

8.3 — dev (the cluster and everything on it)

This is the big one. Fill in the file first:

cd ../dev
cp terraform.tfvars.example terraform.tfvars

Edit terraform.tfvars and set:

project_id            = "YOUR-PROJECT-ID"        # same as step 8.2
cloudflare_api_token  = "YOUR-CLOUDFLARE-TOKEN"
cloudflare_account_id = "YOUR-CLOUDFLARE-ACCOUNT-ID"
cloudflare_zone_id    = "YOUR-CLOUDFLARE-ZONE-ID"
cloudflare_domain     = "YOUR-DOMAIN"            # e.g. example.com
github_repo_owner     = "YOUR-GITHUB-USERNAME"
github_token          = "YOUR-GITHUB-PAT"        # the token from step 6

Then build it. The first build runs in three stages (the cluster has to exist before the tools that talk to it can be set up):

terraform init

# Stage 1 — network, cluster, image registry  (~14 min)
terraform apply -target=module.network -target=module.cluster -target=module.registry -auto-approve

# Stage 2 — Argo CD + the Cloudflare connector  (~5-7 min)
terraform apply -target=module.argocd -target=module.cloudflared -auto-approve

# Stage 3 — the rest  (~1 min)
terraform apply -auto-approve
Only the first time The three stages are needed only on the very first build (or after a full destroy). Normal updates later are just a single terraform apply.

8.4 — vault-bootstrap (passwords for the app)

This connects the cluster to Vault and stores the app's passwords there. Make sure you ran vault login (Step 5) first.

cd ../vault-bootstrap
cp terraform.tfvars.example terraform.tfvars
# edit terraform.tfvars:
#   project_id            = "YOUR-PROJECT-ID"
#   cluster_name          = "prodx-dev-gke"   (leave as-is)
#   postgres_password     = "YOUR-DB-PASSWORD"
#   cloudflare_api_token  = "YOUR-CLOUDFLARE-TOKEN"
#   cloudflare_account_id = "YOUR-CLOUDFLARE-ACCOUNT-ID"
#   cloudflare_zone_id    = "YOUR-CLOUDFLARE-ZONE-ID"
terraform init
terraform apply

8.5 — github-bootstrap (lock down the repos)

This adds branch-protection rules to your three GitHub forks. Do it after the app's CI has run at least once (see 8.6) so the required check already exists.

cd ../github-bootstrap
cp terraform.tfvars.example terraform.tfvars
# edit terraform.tfvars:
#   github_owner = "YOUR-GITHUB-USERNAME"
#   github_token = "YOUR-GITHUB-PAT"
terraform init
terraform apply

8.6 — Ship the app

The infrastructure is up, but the app images still need to be built and pushed. The CI pipeline does this automatically. Trigger it by making any small change on the main branch of your ProdX fork:

cd ProdX
git commit --allow-empty -m "ci: trigger first build"
git push origin main

Or, if you'd rather not touch main, fire both pipelines manually from the Actions tab — or with the gh CLI:

gh workflow run ci-all.yml --ref main --repo YOUR-GITHUB-USERNAME/ProdX

Watch it run in your ProdX repo on GitHub → Actions tab. When it finishes, it has built the website and API images and shipped them to the cluster. Argo CD then deploys them.

How the pieces find each other When envs/dev was applied it automatically published the deploy settings (project, registry, cluster) to your ProdX repo as GitHub Actions variables. The CI pipeline reads those — so you never edit the workflow files by hand, even after a rebuild.

9 · Check it works

From the envs/dev folder:

# Connect kubectl to your new cluster
$(terraform output -raw get_credentials_command)

# The cluster's machines — expect 1-2 lines saying "Ready"
kubectl get nodes

# The app's parts — wait until all show "Running"
kubectl -n prodx get pods

# Argo CD's view of the app — expect "Synced" / "Healthy"
kubectl -n argocd get application prodx

Then open these in your browser (using your domain):

  • https://prodx.YOUR-DOMAIN — the website
  • https://prodxapi.YOUR-DOMAIN/api/metrics — the API (returns some JSON)
  • https://argocd.YOUR-DOMAIN — the Argo CD dashboard
Give it a minute Right after the first build the app pages may show an error for a minute or two while the images download and the pods start. Refresh. If it's still broken after ~5 minutes, see Troubleshooting.

10 · Open Argo CD

Argo CD is the dashboard that shows what's deployed. There are two ways in.

Option A — the public address

Just open https://argocd.YOUR-DOMAIN in your browser.

Option B — a private tunnel from your computer

kubectl -n argocd port-forward svc/argocd-server 8443:443

Leave that running, then open https://localhost:8443.

Getting the password

The username is admin. Get the first-time password with:

kubectl -n argocd get secret argocd-initial-admin-secret -o jsonpath="{.data.password}" | base64 -d

On Windows PowerShell, run this instead:

$p = kubectl -n argocd get secret argocd-initial-admin-secret -o jsonpath="{.data.password}"
[Text.Encoding]::UTF8.GetString([Convert]::FromBase64String($p))
Change the password, then delete the secret After your first login: in the Argo CD UI, top-right user icon → Update Password. Then delete the original so it can't be recovered: 
kubectl -n argocd delete secret argocd-initial-admin-secret

Reading a password out of Vault

The app's database password lives in Vault. The cluster fetches it automatically, but you can look it up yourself any time:

vault kv get secret/prodx/postgres

To change it later: vault kv put secret/prodx/postgres password=NEW-VALUE — the cluster picks up the change within an hour.

11 · Pause & resume (save money)

The cluster is the only part that costs real money. When you're not using the project, pause it — this deletes the cluster but keeps your Google Cloud project, so you can rebuild quickly later.

Pause

cd ProdXTF/envs/dev
terraform destroy

Type yes. Takes ~6 minutes. Your bill drops to almost nothing.

Resume

Rebuild with the same three-stage apply as the first time:

cd ProdXTF/envs/dev
terraform apply -target=module.network -target=module.cluster -target=module.registry -auto-approve
terraform apply -target=module.argocd -target=module.cloudflared -auto-approve
terraform apply -auto-approve

Then push a commit to ProdX main again (Step 8.6) so the app images are shipped to the fresh cluster.

Rebuilds just work A rebuild lands on new internal addresses, but you don't fix anything by hand: terraform apply re-publishes the deploy settings to GitHub and Argo CD automatically. While the project is paused, CI still passes — it just skips the deploy step.

12 · Destroy everything

To remove everything — cluster, project, all of it — destroy the environments in the order below. Note this is not a simple reverse of the build order: vault-bootstrap goes first because its kubernetes_* resources need the cluster to still exist.

# 1. The Vault wiring + secrets (needs the cluster still up)
cd ProdXTF/envs/vault-bootstrap
terraform destroy

# 2. The cluster and everything on it  (~6 min)
cd ../dev
terraform destroy

# 3. The GitHub branch-protection rules
cd ../github-bootstrap
terraform destroy

# 4. The Google Cloud project itself
cd ../bootstrap
terraform destroy

# 5. The HCP Terraform save slots
cd ../tfc-bootstrap
terraform destroy

Type yes at each prompt.

If the dev destroy stalls on the Cloudflare tunnel Right after the cluster goes away, Cloudflare can still see the tunnel as "active" for a few minutes (its edge hasn't noticed the cloudflared pods are gone). Wait ~3 minutes and re-run terraform destroy. If Terraform also reports "Failed to construct REST client" for orphan kubernetes_manifest resources, target the cloudflare resources directly: 
terraform destroy \
  -target=module.cloudflare.cloudflare_zero_trust_tunnel_cloudflared.this \
  -target=module.cloudflare.random_id.tunnel_secret \
  -auto-approve
The 30-day rule — important When you destroy bootstrap, Google deletes the project but reserves its ID for 30 days — you cannot reuse the same project_id during that time. If you rebuild within a month, pick a new project ID (e.g. add a number). This is why pausing with just Step 11 — which keeps the project — is the better "switch off" most of the time.

13 · Troubleshooting

The website shows "502" but Argo CD works

The cluster is healthy, but the app pods can't start. Almost always this means the app images aren't in the registry yet. Fix: run Step 8.6 (push a commit to ProdX main) and wait for the CI run to finish.

A pod is stuck in "ImagePullBackOff"

The pod can't download its image. Check what image it's looking for:

kubectl -n prodx describe pod POD-NAME

It usually means CI hasn't pushed images yet (run Step 8.6), or the last terraform apply in envs/dev didn't finish — re-run it.

The CI pipeline fails at "Auth to GCP"

The cluster's deploy settings haven't been published to GitHub. Run terraform apply in envs/dev; it writes them as GitHub Actions variables. Then re-run the pipeline.

"Error: Requested entity already exists" while applying

Harmless leftover from a recent destroy. Wait a few minutes and run terraform apply again — the project now uses unique names so this clears itself.

terraform apply can't reach the cluster

You're doing a first build with a single terraform apply. Use the three-stage apply from Step 8.3 instead.

14 · Glossary

TermIn plain words
TerraformA tool that builds cloud infrastructure from written instructions, so it's repeatable.
Environment / envA folder of Terraform code you build separately (this project has five).
StateTerraform's memory of what it has already built. Stored online in HCP Terraform.
KubernetesSoftware that runs and manages your app's containers across servers.
GKEGoogle's managed Kubernetes — the cluster this project runs on.
ClusterThe group of servers Kubernetes manages.
PodThe smallest running unit in Kubernetes — basically one running piece of the app.
Container imageA packaged, ready-to-run copy of a program. CI builds these.
Artifact RegistryGoogle's private storage for your container images.
Helm chartA template describing what to deploy into the cluster (this is ProdXCD).
Argo CDA tool that keeps the cluster matching the Helm chart in Git, automatically.
GitOpsThe idea that Git is the single source of truth for what's deployed.
CI / CDAutomation that builds, tests and ships your code when you push to GitHub.
Cloudflare TunnelA secure connection that exposes the app on real HTTPS URLs without a public IP.
VaultA secure store for passwords and tokens (secrets).
Workload Identity FederationA way for GitHub's CI to log in to Google Cloud without a stored password.
PATPersonal Access Token — a password-like key that lets tools use your GitHub account.
