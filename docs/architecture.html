ProdX — Architecture & Components

What every piece of this project is, why it's here, how they connect to each other, and how the whole thing is secured end-to-end.

The shape, in one sentence Code lives in GitHub; GitHub Actions builds and ships images to a private Artifact Registry; Argo CD running on a GKE cluster syncs them according to a Helm chart; the cluster pulls runtime secrets from Vault via ESO; and traffic reaches it over a Cloudflare Tunnel — with no long-lived credentials anywhere.

2 · The three repositories

The project deliberately splits into three git repositories. Each one has a single job and a different reason to change. Mixing them in one repo is the classic "monorepo soup" mistake that makes blast radius huge and reviews painful.

RepoHoldsChanges when…Affects
ProdXThe app: FastAPI backend + React frontend + Dockerfiles + CI workflows + this documentation site. A developer ships a feature or bug fix. New container images get built and rolled out.
ProdXTFTerraform code for every cloud resource: GCP project, GKE, VPC, Artifact Registry, WIF, Cloudflare tunnel, Vault wiring, GitHub branch protection. The infrastructure shape changes (new module, sizing, regions, IAM). Cloud resources get created / changed / destroyed.
ProdXCDThe Helm chart describing what runs inside the cluster: Deployments, Services, ConfigMaps, ExternalSecrets, ClusterSecretStore. Workload shape changes (replicas, env vars, resource limits, new K8s objects). Argo CD reconciles the cluster to match.
Why this matters for security and operations Application code can be reviewed by app devs; infra by SREs; cluster manifests by platform engineers. Each repo gets its own branch-protection rules, its own required reviewers, its own CI. A bad pull request in one cannot directly compromise the others.

3 · GitHub

Source of truthGitOps

What: Git hosting + collaboration platform. Holds the three repos, runs Actions CI, hosts releases and Dependabot PRs.

Why: Git is the immutable record of what should be deployed — both the application code and the desired cluster state. Argo CD literally reads from GitHub to know what to run; nothing is "configured by clicking in a dashboard."

How it connects:

  • Developer pushes → GitHub stores commits, runs Actions workflows.
  • GitHub OIDC issues short-lived JWTs to CI jobs → Google Cloud accepts them via WIF.
  • Argo CD polls the ProdXCD repo (HTTPS, read-only) for the Helm chart.
  • Terraform's github provider sets branch protection + repo variables.

Security

  • Branch protection on main for all three repos: no force-push, no deletion, required CI check, optional PR reviews. Managed in Terraform — not click-driven.
  • Fine-grained Personal Access Tokens with the minimum scopes; no classic admin keys.
  • Forks can't push images: the WIF provider's attribute_condition rejects OIDC tokens whose repository claim doesn't match exactly OWNER/REPO.

4 · GitHub Actions (CI)

CI

What: The build & ship pipeline. Two main workflows (ci-backend.yml, ci-frontend.yml) plus skip-stubs and an optional Cursor review.

Why: Every change goes through the same scripted gate. Humans don't push to :latest; the workflow does, with a fresh OIDC token each run.

On a pull request it runs:

  • Lint (ruff) + tests (pytest) + dependency audit (pip-audit / npm audit).
  • Docker build (multi-stage, OCI-labelled).
  • Trivy scan of the built image — fails the build on any fixable CRITICAL CVE.
  • The Cursor agent posts a review comment (optional, non-blocking).

On a push to main it additionally:

  • Authenticates to Google Cloud via Workload Identity Federation — no stored key.
  • Pushes the image to Artifact Registry under main, sha-XXXX, and latest.
  • Runs kubectl rollout restart against the cluster so pods re-pull :latest.

Path filters keep things efficient: a backend-only change only fires the backend pipeline; a frontend-only change only fires the frontend pipeline. The "skip" stubs satisfy branch-protection's required check when the other side is untouched.

Deploy coordinates are NOT hardcoded. GCP_PROJECT_ID, GCP_WIF_PROVIDER, GCP_WIF_SA, GAR_HOST, image repos, cluster, zone — all come from GitHub Actions repository variables written by ProdXTF/envs/dev. The workflow reads them as ${{ vars.* }}; a destroy/recreate just re-publishes them. While the environment is destroyed, the variables are absent and the publish/deploy steps skip cleanly — CI stays green on build + scan.

Security

  • No long-lived service-account key — WIF mints a 1-hour token per run.
  • Trivy gates merges on CRITICAL CVEs with a known fix.
  • pr-review.yml is forks-blocked: forked PRs don't get repo secrets and the agent skips them.

5 · Cursor PR review (AI second opinion)

CI

What: An automated AI code review that posts a markdown comment on every same-repo pull request, flagging bugs / regressions / unclear design.

Why: A cheap extra pair of eyes. It does not block merges (it's intentionally off the required-checks path); a human reviewer is still the gate.

How it connects: pr-review.yml installs cursor-agent, generates a three-dot diff of the PR, and asks the agent for a concise review. It uses the CURSOR GitHub Actions secret — set by Terraform from the Vault path secret/prodx/cursor when set_cursor_secret = true.

6 · Dependabot

CISupply-chain

What: GitHub's bot that watches dependency manifests (requirements.txt, package.json, GitHub Action versions) and opens PRs when a newer version is available — especially when a known CVE is fixed upstream.

Why: Stale dependencies are the most common breach vector. Dependabot turns "we should update someday" into "there's a PR ready and CI proved it builds." Combined with Trivy's CRITICAL-CVE gate, the project stays patched almost by itself.

How it connects: Config lives at ProdX/.github/dependabot.yml. Dependabot opens a PR → that PR runs the same Actions CI (build + Trivy scan) → if green, a human merges. Every recent commit titled chore(deps): bump … in the log is one of these.

7 · HCP Terraform

IaC

What: HashiCorp's hosted Terraform state backend. We use it for two workspaces (prodx-bootstrap and prodx-dev) with Execution Mode = Local: state lives in HCP, but terraform apply runs on your laptop with your local credentials.

Why local execution? Remote runs would need HCP's servers to hold your gcloud, vault and kubectl tokens — that's exactly the secret sprawl we're avoiding. Local exec keeps credentials on your laptop and uses HCP only for the (encrypted, version-locked, team-safe) state file.

Why state in HCP at all? A tfstate committed to Git is a recipe for losing or leaking state. HCP locks it, versions it, and lets multiple people collaborate later without stomping each other.

Sub-workspaces with local state: tfc-bootstrap, github-bootstrap and vault-bootstrap are one-time bootstraps that keep state on disk — they configure the things HCP and the rest depend on, so they can't depend on them in turn.

8 · Google Cloud project

Cloud

What: A dedicated GCP project (e.g. prodx-dev3) that owns the cluster, network, Artifact Registry repo, and service accounts. Created by envs/bootstrap.

Why an isolated project? A project is GCP's blast-radius boundary — IAM, quotas, billing and auditing are all per-project. Anything wrong in this project (a runaway pod, a leaked SA key, a billing surprise) is contained to the project; nothing else in your Google account is touched.

How it connects: Terraform turns on exactly the APIs we need (compute, container, IAM, Artifact Registry, IAM Credentials, STS) and grants narrowly-scoped roles. The 30-day project-id reservation on delete is why destroy/recreate picks a fresh id.

9 · GKE cluster (Kubernetes)

Cluster

What: A zonal Google Kubernetes Engine cluster — 1 to 3 small Spot VMs, autoscaling, with Workload Identity enabled at the cluster level.

Why GKE specifically?

  • Managed control plane — no etcd to babysit.
  • Cluster-level Workload Identity ties K8s ServiceAccounts to GCP service accounts without static JSON keys (used here for pull access from the node SA to Artifact Registry).
  • Spot VMs are 60–80% cheaper than on-demand and fine for a dev / capstone cluster — autoscaling masks preemptions.

What runs inside:

  • argocd namespace — the Argo CD installation.
  • prodx namespace — the application (api, web, postgres, redis).
  • external-secrets namespace — ESO controller.
  • cloudflared namespace — the tunnel connector pods (outbound only).
  • vault-auth namespace — a reviewer ServiceAccount Vault uses to validate other pods' JWTs.

Security

  • Private VPC; no public IPs on nodes; outbound via Cloud NAT.
  • No public LoadBalancer — Services are ClusterIP; ingress is the Cloudflare Tunnel (outbound).
  • Per-repo Artifact Registry IAM (the node SA gets read on this repo only).

10 · Artifact Registry

Cloud

What: Google's private container registry. One repo (prodx) holds two images (prodx-api, prodx-web) tagged with main, sha-XXXX and latest.

Why private (over Docker Hub)? Replaces the last long-lived secret in CI (Docker Hub PAT). Also keeps the images private — they ship to your cluster, not to the public Internet.

How it connects:

  • CI push: the gha-prodx-publisher service account has artifactregistry.writer on the repo. CI impersonates it via WIF.
  • GKE pull: the cluster's default compute SA has artifactregistry.reader on the same repo. Pulls require no imagePullSecret.
  • Cleanup policy: untagged layers older than 30 days are GC'd; tagged images stay forever for incident forensics.

The image path embeds the GCP project id, which changes on every destroy/recreate — that's why ProdXCD/values.yaml only carries a placeholder and the Argo CD Application overrides it with a Helm parameter built from the live registry module.

11 · Workload Identity Federation (WIF)

Security

What: The trust binding that lets GitHub Actions get a short-lived GCP token without any stored service-account key.

Why: A JSON SA key in secrets.GCP_SA_KEY is a long-lived credential — a leak compromises every run forever. WIF gives each CI run a 5-minute OIDC JWT that GCP exchanges for a 1-hour access token. There's no key to steal.

How the handshake works:

GitHub runner ─(1) requests OIDC token─▶ token.actions.githubusercontent.com
                                            (signs a JWT: sub, repository, ref, workflow)
GitHub runner ─(2) passes JWT────────────▶ GCP STS (workload identity pool provider)
                                            ▲
                                            │ attribute_condition CEL:
                                            │   assertion.repository == "OWNER/REPO"
                                            │ → rejects forks and other repos
                                            ▼
GCP STS      ─(3) returns federated token─▶ runner
GCP STS      ─(4) impersonates SA──────────▶ gha-prodx-publisher@PROJECT.iam.gserviceaccount.com
                                              (1-hour access token; only artifactregistry.writer
                                               on the prodx repo + container.developer on the project)

Reference impl: ProdXTF/modules/registry/main.tf. The pool id now carries a random suffix per generation so a destroy/recreate never collides with GCP's 30-day soft-delete.

Security

  • The single biggest line in the module is the attribute_condition — without it, any GitHub Actions workflow on the entire planet could request a token from this provider.
  • The publisher SA has writer on one Artifact Registry repo, not the project. Worst-case blast radius is "1 hour of pushes to prodx."
  • Revocation = remove an IAM binding. No "rotate the key everywhere."

12 · Argo CD

GitOps

What: A controller running in the cluster that continuously reconciles what's deployed against a Helm chart in a Git repo. The Git repo is the source of truth; if you change the cluster by hand, Argo CD changes it back.

Why GitOps?

  • Audit trail. Every change is a Git commit. No "who changed this in kubectl at 3am?"
  • Repeatability. Rebuild the cluster, point Argo at the same repo, get the same workloads back.
  • Drift detection. A misconfigured pod or a manual fix is visible in the UI as OutOfSync until it's reconciled.
  • Self-healing. Delete a pod by accident — Argo recreates it.

How it's wired: Terraform installs the Argo Helm chart and creates a single root Application CR pointing at ProdXCD/charts/prodx. From there, Argo CD owns the cluster's state.

Image repo override: Argo's root Application injects the current GCP project's Artifact Registry path as a Helm parameter, so the chart in Git stays project-agnostic.

13 · Helm

Cluster

What: A templating system for Kubernetes manifests. The ProdXCD repo is one Helm chart; values.yaml is its tunables (replicas, images, resource limits…).

Why: Helm gives us one place to bump the API replica count, change a CORS origin, swap a postgres image — without editing six raw YAML files. Argo CD renders the chart on every sync.

14 · Cloudflare Tunnel

EdgeSecurity

What: A reverse proxy that exposes in-cluster services on real HTTPS URLs without giving the cluster any public IP or open inbound port. The cloudflared pod inside the cluster dials out to Cloudflare's edge; the edge relays incoming HTTPS requests down that outbound connection.

Why:

  • Zero managed certificates — Cloudflare handles TLS at the edge.
  • Zero attack surface — the cluster doesn't listen on the public Internet at all.
  • No GCP LoadBalancer cost (those are $18+/mo each).

How it connects: Terraform creates the tunnel, three proxied DNS CNAMEs (argocd, prodx, prodxapi on your apex domain) and the ingress config (which hostname routes to which in-cluster Service). The tunnel run-token is generated by Terraform and mounted into the cloudflared pod as a K8s Secret.

15 · HashiCorp Vault

Security

What: A separate Vault server (this project uses vault.YOUR-DOMAIN) that stores every runtime secret: the Postgres password, the Cloudflare API token, the GitHub PAT, the Cursor key.

Why a dedicated secret store?

  • Rotation. Change the Postgres password in Vault — the cluster picks it up on the next ESO refresh. No rebuild, no Git commit, no plaintext leak.
  • Auditable reads. Vault logs every secret access.
  • Encryption. Secrets are sealed at rest; only auth'd identities can read.

How it connects:

  • Terraform on your laptop reads tokens with data.vault_kv_secret_v2 using your vault login token. Used at apply time only.
  • ESO inside the cluster authenticates using the K8s Service Account JWT and a Vault role bound to the prodx-app-read policy.

Trust is set up by envs/vault-bootstrap: it enables Vault's Kubernetes auth method, points it at the cluster's API server, and creates the prodx-eso role.

16 · External Secrets Operator (ESO)

ClusterSecurity

What: A Kubernetes controller that pulls secrets from an external store (Vault, in our case) and materializes them as native K8s Secret objects.

Why not just embed secrets in the Helm chart? Because anything in a chart sits in Git. ESO keeps the chart secret-free — it just contains an ExternalSecret CR that says "pull secret/prodx/postgres from Vault and create a K8s Secret named postgres-credentials." The actual value never lives in Git.

How it connects:

  ESO pod ───(1) reads its own SA JWT─────────────▶ Kubernetes API
  ESO pod ───(2) sends JWT to /v1/auth/kubernetes─▶ Vault
  Vault    ───(3) TokenReview against cluster API─▶ Kubernetes API
                                                     │
                                                     ▼ valid → role: prodx-eso
  Vault    ───(4) returns short-lived token───────▶ ESO pod
  ESO pod  ───(5) reads secret/prodx/postgres─────▶ Vault
  ESO pod  ───(6) creates/updates K8s Secret──────▶ Kubernetes API
  Postgres ───(7) mounts Secret as env vars────── done.

Refresh interval is 1 hour, so a rotation in Vault propagates within the hour with no pod restart.

17 · Trivy

Security

What: A vulnerability scanner that inspects a container image's OS packages and language dependencies for known CVEs.

Why: The package-audit step (pip-audit, npm audit) only sees what's in your direct dependencies. Trivy also catches CVEs in the base image — the FastAPI Dockerfile inherits from python:3.12-slim, the nginx image from nginx:alpine, and base images get new CVEs constantly.

How it's gated: CI runs trivy image with severity: CRITICAL and ignore-unfixed: true. A merge is blocked only when there's a fixable CRITICAL CVE — i.e. the upstream package has a patch and we just haven't picked it up yet. This avoids the "every PR fails because alpine has 700 unfixed CVEs" trap.

     │
     │       (the steps below only run on push to main / manual dispatch,
     │        AND only when GCP_PROJECT_ID is published — i.e. the

     │
     ├─ 🔑  google-github-actions/auth (WIF — no stored key)
     ├─ 🐳  gcloud auth configure-docker

     └─ 🔁  kubectl rollout restart deploy/prodx-{api,web}
                       │
                       ▼
                   GKE cluster picks up the new image
                   (imagePullPolicy: Always re-pulls :latest)

Every step is conditional, and the conditions are why the pipeline is robust:

  • PR vs push: quality checks run on PRs and manual dispatches; the publish/deploy steps don't.
  • Env present vs absent: publish/deploy ALSO gate on vars.GCP_PROJECT_ID != ''. While Terraform has destroyed the env, those variables are empty and the steps skip — so CI stays green on build + scan even when there's no cluster.
  • Backend vs frontend: path filters route a touch in backend/** to only the backend workflow. The companion skip-stubs satisfy branch protection's required check name.

19 · The CD pipeline — from image to running pod

There are two complementary delivery loops. CI does the push; Argo CD does the pull.

  (1) Image-tag loop (per code change)
      CI pushes prodx-api:latest, :sha-XXX  ─▶ Artifact Registry
      CI runs `kubectl rollout restart deploy/prodx-api`
         │
         ▼
      Kubernetes creates new pods; imagePullPolicy: Always
      re-pulls :latest, which now resolves to the new digest.

  (2) Chart-state loop (per chart change in ProdXCD)
      Edit ProdXCD/values.yaml or template, push to main
         │
         ▼
      Argo CD polls the repo, detects drift, runs `helm template`
         │
         ▼
      Argo CD applies the rendered manifests to the cluster.
      ApplicationSet.status flips Synced/Healthy.

The two loops are independent on purpose: an app developer ships code without ever touching cluster YAML, and a platform engineer changes cluster shape without ever rebuilding an image.

22 · Security at maximum — what makes this hard to attack

Identity & access

  • No long-lived cloud keys. WIF + GitHub OIDC for CI; Workload Identity for nodes; Vault-issued K8s auth tokens for ESO.
  • Repo-scoped trust. WIF rejects OIDC tokens whose repository claim doesn't match — forks cannot push.
  • Least-privilege IAM. Publisher SA: writer on one AR repo. Node SA: reader on one AR repo. No project-wide grants beyond container.developer (needed for rollout restart).
  • Fine-grained PATs. The GitHub token has Variables, Administration, Actions, Metadata — not repo god-mode.

Supply chain

  • Trivy CRITICAL gate on every image build.
  • Dependabot raises PRs the moment an upstream fix lands.
  • Private registry. Images never go to a public hub.
  • Pinned provider + chart versions in Terraform.
  • Cursor PR review as a non-blocking second opinion.

Network

  • No public IP on the cluster. Inbound traffic only via the Cloudflare Tunnel (outbound).
  • TLS terminated at Cloudflare — no certificate management on the cluster.
  • ClusterIP services only; nothing of type LoadBalancer or NodePort exposed.
  • Private VPC + Cloud NAT for outbound from the nodes.

State & secrets

  • State in HCP, locked + versioned; never in Git.
  • Secrets in Vault, never in Git. *.tfvars is git-ignored.
  • ESO reads with a 1h-TTL Vault token, per-namespace, per-SA.
  • Rotation by editing Vault — no commits, no re-deploys.

Code & review

  • Branch protection on main for all three repos.
  • Required CI check (🚀 Pipeline) before merge.
  • No force-push, no branch deletion.
  • PR-review.yml is fork-gated so a malicious fork can't exfiltrate secrets via a workflow change.

Blast radius

  • Dedicated GCP project — billing, IAM, audit all contained.
  • Per-repo IAM bindings, not project-wide.
  • Cluster-wide deletes require both deletion-protection off AND a Terraform action.
  • Vault secrets soft-deleted (KV-v2 versions); recoverable.
What we deliberately did NOT do (and why it's OK here) No mTLS service mesh, no OPA policies, no signed images (cosign), no node attestation, no private GKE control-plane endpoint. These are real and reasonable for prod — but this is a capstone / dev setup where they'd add complexity without an attacker who'd notice. The architecture is structured so any of them can be added later without rewriting what's here.
